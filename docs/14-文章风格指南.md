# 文章风格指南

> 本项目支持 7 种风格的技术博客，根据内容性质选择合适的风格。

## 风格总览

| 风格 ID | 名称 | 核心特征 | 典型场景 |
|---------|------|---------|---------|
| `zhihu` | 知乎专栏风 | 强烈第一人称、主观判断、个人经验 | 踩坑分享、技术选型 |
| `paper` | 论文风 | 客观中立、引用真实案例、严谨 | 技术深度分析、研究综述 |
| `popular_science` | 深度科普风 | 类比驱动、面向非技术读者 | 少数派/36氪深度稿 |
| `newsletter` | Newsletter 风 | 简短精炼、要点+链接 | 邮件周报、技术快讯 |
| `tutorial` | 教程/实战风 | step-by-step、代码示例 | 开发者教程、上手指南 |
| `security` | 安全分析风 | 攻防视角、漏洞分析 | 安全研究报告 |
| `toolbox` | 工具推荐风 | 结构化评测、优劣对比 | 工具横评、选型指南 |

## 风格详细对比

| 维度 | 知乎专栏风 | 论文风 | 深度科普风 | Newsletter 风 | 教程风 | 安全分析风 | 工具推荐风 |
|------|-----------|--------|-----------|--------------|--------|-----------|-----------|
| 人称 | 强烈"我" | 第三人称 | "你" | 无人称 | "你/我们" | 第三人称 | 朋友口吻 |
| 立场 | 主观判断 | 客观中立 | 温和引导 | 客观简洁 | 实操引导 | 严肃专业 | 平衡评价 |
| 长度 | 2000-4000 | 2500-5000 | 2000-3500 | 800-1500 | 2000-5000 | 2000-4000 | 2000-3500 |
| 代码 | 少量 | 适量 | 禁止 | 禁止 | 大量 | 适量 | 少量 |
| 数据要求 | 个人经验 | 必须标注来源 | 用类比替代 | 具体数字 | 可运行代码 | CVE/官方来源 | 版本/价格 |

## 如何选择风格

| 场景 | 推荐风格 | 原因 |
|------|---------|------|
| 分享个人踩坑经验 | 知乎专栏风 | 更有感染力，读者易共情 |
| 技术选型决策分享 | 知乎专栏风 | 主观判断更有参考价值 |
| 方法原理深度分析 | 论文风 | 更客观、更专业 |
| 多方案横向对比 | 论文风 | 结构化对比更清晰 |
| 没有实践经验时 | 论文风 | 避免虚构个人经历 |
| 向非技术人员解释 | 深度科普风 | 类比驱动，无门槛 |
| 技术周报/快讯 | Newsletter 风 | 信息密度高，快速消费 |
| 教别人怎么做 | 教程风 | step-by-step 可跟做 |
| 漏洞/安全事件分析 | 安全分析风 | 攻防视角，严谨专业 |
| 推荐/评测工具 | 工具推荐风 | 结构化对比，场景化推荐 |

## 深度科普风的核心原则

### 1. 类比驱动
每个技术概念必须用日常生活类比解释：
- "数据库索引就像图书馆的目录卡"
- "API 就像餐厅的服务员——你不需要知道厨房怎么做菜"
- "缓存就像你把常用电话号码写在便签纸上"

### 2. 场景引入
不要上来就定义，用场景开头：
- ❌ "Kubernetes 是一个容器编排系统"
- ✅ "想象你经营一家快餐连锁店，每到午餐高峰，你需要临时增开几个窗口..."

### 3. 禁止代码
不出现任何代码块。需要解释技术过程时，用自然语言描述。

## Newsletter 风的核心原则

### 1. 极致精简
每条 2-3 句说清核心，不展开论证。读者用 2 分钟扫完全部内容。

### 2. 信息密度
每句都有信息量。删掉任何一句都会丢失信息。

### 3. 条目格式
```
**加粗标题**（≤ 15 字）
核心说明 2-3 句。具体数字/版本/日期。
→ [相关链接](url)
---
```

## 教程/实战风的核心原则

### 1. 可跟做
每一步都要能实际操作。代码必须可运行，不用省略号替代关键逻辑。

### 2. 前置条件明确
开头说清：需要什么环境、什么版本、什么依赖。

### 3. 检查点
每 2-3 步后有验证："到这里，你应该能看到..."

### 4. 踩坑标注
```
> ⚠️ 注意：如果遇到 X 错误，检查 Y 是否正确配置。
```

## 安全分析风的核心原则

### 1. 事实与推测分离
已确认的事实直接陈述，推测用"据分析"/"推测认为"标记。

### 2. 具体数据
版本号、CVE 编号、影响范围、CVSS 评分——越具体越好。

### 3. 可操作的缓解措施
不要"注意安全"这种废话，要具体的修复命令/配置变更。

## 工具推荐风的核心原则

### 1. 优劣都说
每个工具都有优势和劣势。不回避缺点才有公信力。

### 2. 场景化推荐
结尾给出："如果你是前端工程师，推荐 X；如果你是全栈，推荐 Y"。

### 3. 数据具体
版本号、价格（免费/付费/开源）、支持的平台。

## 论文风的核心原则

### 1. 不要虚构个人经历

**❌ 错误示例**：
```
我在生产环境中使用了 BERT，发现延迟很高。我们团队做了实验，
发现使用蒸馏后的模型，性能提升了 30%。
```

**✅ 正确示例**：
```
根据 Google 2024 年的技术博客，BERT 在生产环境中的平均
推理延迟为 50ms。Google 团队通过知识蒸馏技术，将模型参数
压缩到原来的 40%，推理延迟降低至 15ms，性能提升约 70%。
```

### 2. 引用真实案例

**必须标注来源**：
- 论文引用：论文标题 + 作者 + 年份
- 技术博客：公司名 + 文章标题
- 官方报告：公司名 + 报告名称 + 年份

**常用案例来源**：
- Google Research Blog
- OpenAI Blog
- Microsoft Azure Blog
- Netflix Tech Blog
- Uber Engineering Blog
- 论文数据库（arXiv、ACL、ICML 等）

### 3. 客观表述

**知乎专栏风**（主观）：
```
我觉得这个方法不靠谱。我们试了之后发现效果很差。
```

**论文风**（客观）：
```
根据 Smith 等人 (2023) 的研究，该方法在低资源场景下
性能下降显著，准确率从 85% 降至 62%。
```

### 4. 数据标注

**所有数据必须标注来源**：
```
❌ 模型推理速度提升了 30%
✅ 根据 OpenAI 官方报告 (2024)，GPT-4 Turbo 相比 GPT-4，
   推理速度提升了约 3 倍（从 30 tokens/s 提升至 90 tokens/s）
```

## 如何获取真实案例

### 1. 官方技术博客

- **Google**: https://blog.google/technology/
- **OpenAI**: https://openai.com/blog
- **Microsoft**: https://azure.microsoft.com/en-us/blog/
- **Netflix**: https://netflixtechblog.com/
- **Uber**: https://www.uber.com/blog/engineering/

### 2. 学术论文

- **arXiv**: https://arxiv.org/
- **Google Scholar**: https://scholar.google.com/
- **Semantic Scholar**: https://www.semanticscholar.org/

### 3. 公司技术报告

- 各大公司的年度技术报告
- 开源项目的官方文档
- 技术会议演讲（如 Google I/O、WWDC）

## 混合风格（慎用）

某些文章可以混合使用两种风格：
- 开头用论文风（客观背景）
- 中间用知乎风（个人经验）
- 结尾用论文风（客观建议）

**注意**：混合风格容易导致风格不一致，建议优先选择单一风格。

## 检查清单

### 知乎专栏风检查清单
- [ ] 至少 3 次出现"我"或"我们"
- [ ] 至少 1 处个人判断
- [ ] 句子长短交替，节奏快
- [ ] 段落 ≤ 4 句

### 论文风检查清单
- [ ] 所有数据都有来源标注
- [ ] 没有虚构个人经历
- [ ] 使用客观表述（"研究表明"、"数据显示"）
- [ ] 至少 1 处真实案例引用
- [ ] 所有专业术语首次出现有解释
